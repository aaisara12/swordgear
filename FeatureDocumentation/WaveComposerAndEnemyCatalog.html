SwordGear · M6 Encounter System

Wave Composer & Enemy Catalog

How runtime combat encounters are built from a threat budget, catalog costs, themes, and roles — and how to tune them for denser waves, elemental themes, or later-block difficulty.

1. Overview

Combat waves are not hand-authored ScriptableObject wave lists anymore (those still exist as legacy). When the player enters a combat step, the game builds a runtime CombatEncounter: a list of waves, each containing spawn specs (archetypeId + optional elite flag).

Two assets drive everything
  • EnemyCatalog.asset — who can spawn (20 archetypes), each with a threat cost and prefab.
  • WaveComposerSettings.asset — how much to spend per wave, how many waves, theme/role weights, block scaling, caps.

Determinism: same run seed + same step index → identical encounter composition (spawn positions in the room still use Unity random).

2. Assets & where to tune

3. Run structure (how context is derived)

The linear rail is repeating blocks of Combat → Combat → Combat → Upgrade.

TermMeaning
blockIndex0 for first loop, 1 after first upgrade, etc.
combatIndexInBlock0, 1, or 2 — which of the three fights in the block.
globalStepIndexAbsolute step on the rail (0, 1, 2, 4, 5… skipping upgrades for combat context).
CombinedSeed()(runSeed * 397) ^ globalStepIndex — feeds System.Random for composition.

Profile lookup uses combatIndexInBlock (which of the three combat profiles). Block scaling uses blockIndex.

4. Enemy Catalog (deep dive)

20 archetypes = 5 roles × 4 elements.

4.1 Roles

RoleBehaviour (short)Typical cost band
MeleeChase + melee~10–14
RangedStrafe + projectile~12–16
BeamSniperTelegraphed laser~18–22
ShotgunPellet spread~14–18
TurretStationary shooter~16+

4.2 Per-archetype fields

FieldWhat it does
id Stable string the composer stores on spawn specs (e.g. melee_fire). LevelLoader looks this up to get the prefab.
prefab The enemy GameObject to Instantiate.
element / role Used when filtering candidates after role/element rolls.
baseThreatCost Fixed cost subtracted from the per-wave budget when that enemy is chosen. Does not scale with block index.
applyElementKnobsAtSpawn If true (new archetypes), multiply stats by catalog ElementStatKnobs at spawn. Legacy melee/ranged usually false (stats baked into prefab).

4.3 Element stat knobs (spawn-time, not composition)

For archetypes with applyElementKnobsAtSpawn, defaults roughly:

  • Fire — more damage/speed, aggressive.
  • Ice — much more HP/damage, slower.
  • Lightning — fast, weak hits, short charge.
  • Physical — identity (1×).

These stack on top of the shared block HP/damage multipliers from WaveComposerSettings.

Cost vs difficulty Raising an enemy’s baseThreatCost makes it “rarer” under a fixed budget (fewer of them fit). It does not make that enemy tankier. Tankiness comes from prefab stats, element knobs, block HP scale, and elite multipliers.

5. Wave Composer Settings (every field)

5.1 Combat profiles (3 slots)

Index 0 / 1 / 2 = first / second / third combat in a block.

FieldMeaning
threatBudget Per wave. Each wave in that fight independently spends this much (then × block threat scale). Example: budget 80 with 3 waves ≈ up to ~240 total cost spent if every wave fills.
minWaves / maxWaves Inclusive random wave count for that fight.
guaranteeElite If true, at least one elite in the fight (preferably first spawn of the last wave).

Within a block, HP and damage are identical across the three profiles. Use budget + wave count (+ elite) for within-block pacing.

5.2 Later-block scaling

Formula: value × (1 + blockIndex × scale)

FieldApplies to
blockThreatScalePer-wave threat budget (more enemies later, until caps).
blockHpScaleShared HP mult for all three fights in that block.
blockDamageScaleShared damage mult for all three fights in that block.
 
Block 0: HP = 1.0, budget = base
Block 1 (scale 0.25 HP / 0.35 threat): HP = 1.25, budget = base × 1.35
Block 2: HP = 1.5, budget = base × 1.7

5.3 Theme weights

One array per combat slot. Order is always:

  1. Mixed
  2. Physical
  3. Fire
  4. Ice
  5. Lightning

Relative weights only (do not need to sum to 100). Theme is picked once per fight.

5.4 Role weights

Order is always:

  1. Melee
  2. Ranged
  3. BeamSniper
  4. Shotgun
  5. Turret

Rolled independently for every enemy spawn.

5.5 Composition limits

FieldMeaning
maxEnemiesPerWaveHard cap. Extra budget past this does nothing for count.
minEnemiesPerWaveAlways at least this many (can overspend slightly).
delayAfterClearSeconds after a wave dies before the next starts.
themedElementBias If theme is not Mixed: with probability bias, force the themed element; otherwise roll any of the 4 elements with equal probability (including the themed one again). So effective themed rate is a bit higher than the bias alone.

6. End-to-end pipeline

  1. Player enters combat arena
    NodeStarter → RunManager.BuildBlueprintForCurrentStep().
  2. Build encounter context
    From current RunStep: seed, block, combat index.
  3. Compose
    EncounterBuilder.Build → WaveComposer.Compose(catalog, settings) returns a CombatEncounter.
  4. Blueprint
    LevelBlueprint holds arena layout + that encounter (legacy wave list empty).
  5. Load room
    LevelLoader instantiates arena prefab, spawns player at PlayerSpawnMarker.
  6. Play waves
    For each composed wave: wait delay → instantiate catalog prefabs at random EnemySpawnPoints → apply modifiers → spawn presentation → fight until clear.

7. Compose algorithm (step-by-step)

This is the exact logic in WaveComposer.Compose.

7.1 Fight-level setup

  1. Create System.Random(CombinedSeed()).
  2. Resolve combat profile (budget/waves/elite) with block scaling for threat + shared HP/damage.
  3. Pick one theme from that combat’s theme weights.
  4. Pick wave count in [minWaves, maxWaves].
  5. Assign each wave the full per-wave budget (no splitting across waves).

7.2 Filling one wave

  1. Set remaining = budgetPerWave.
  2. Loop while under min count, or (remaining ≥ cheapest catalog cost AND count < max):
    • Roll role from role weights.
    • Roll element (theme bias rules above).
    • Find catalog match: prefer exact role+element; else same role any element; else any archetype.
    • If last wave + guarantee elite + first spawn of that wave → mark elite.
    • If already at min and can’t afford usefully / remaining ≤ 0 → stop.
    • Else append spawn spec and subtract baseThreatCost from remaining (slight overspend allowed).
  3. If wave empty → force one random enemy (elite if needed).

7.3 After all waves

If elite was required but never placed, force the first spawn of the last wave to elite.

Package theme, waves, budget, spent, difficulty modifiers; log [WaveComposer] ….

8. Arena spawn path (after composition)

  1. Announce wave number (banner shows WAVE N only).
  2. Wait DelayAfterClear.
  3. For each ComposedSpawnSpec: lookup prefab by id → Instantiate at random spawn point.
  4. Build modifiers:
    • Shared encounter HP/damage (from block scales)
    • × element knobs if applyElementKnobsAtSpawn
    • × elite presentation if IsElite
  5. EnemySpawnPresentation plays spawn clip; AI disabled until done.

9. Worked examples

Example A — Combat 1, block 0, budget 80, 2 waves

Per-wave budget: 80
Cheapest melee ~10 → theoretical max ~8 enemies/wave before maxEnemiesPerWave
With maxEnemiesPerWave = 6 → each wave stops at 6 even if budget remains

Possible wave 1 spend:
  melee_physical (10) + ranged_fire (13) + shotgun_physical (14) + turret_fire (16) + …
  until remaining < cheapest or count == 6

Example B — Same profile, but 3 waves

Total threat capacity ≈ 80 × 3 = 240 (before caps)
Previously (old split): wave1 only got ~13 → often 1 enemy
Now: every wave gets a full 80

Example C — Fire theme, bias 0.72

Theme roll → Fire
Each spawn:
  72% → force Fire
  28% → equal among Physical/Fire/Ice/Lightning (25% each of that 28%)

Approx P(Fire) ≈ 0.72 + 0.28×0.25 = 0.79
Not “72% Fire / 28% non-Fire only”.

Example D — Block 1 with threatScale 0.35, hpScale 0.25

Base per-wave budget 110 → effective 110 × 1.35 = 148.5
All three fights in block 1 share HP × 1.25 and damage × (1 + blockDamageScale)
Enemy costs still 10–22 — more enemies fit until maxEnemiesPerWave

10. Tuning recipes (goal → knobs)

Make early fights denser

Raise threatBudget on combat profile 0. Optionally raise maxEnemiesPerWave if you hit the cap. Lower melee/ranged costs in the catalog if you want more bodies without raising budget.

Make combat 3 feel like a climax without tankier stats

Higher budget + higher min/max waves + guaranteeElite = true on profile 2. Keep HP/damage flat (already). Push role weights toward BeamSniper / Shotgun / Turret.

Mostly melee / tutorial-ish opening

roleWeightsCombat0 ≈ high Melee, some Ranged, near-zero sniper/shotgun/turret. Theme weights toward Mixed/Physical.

Strong elemental “Fire Assault” nights

Raise Fire weight in theme arrays; raise themedElementBias toward 0.85–0.95. Remember miss-rolls can still pick Fire again.

Later blocks harder without packing the arena

Keep maxEnemiesPerWave modest. Raise blockHpScale / blockDamageScale. Optionally lower blockThreatScale so count doesn’t explode.

Later blocks denser (more enemies)

Raise blockThreatScale and/or maxEnemiesPerWave. Costs stay fixed, so budget growth directly buys more spawns until the cap.

Make snipers “expensive” so they appear less often under budget pressure

Raise baseThreatCost on beam sniper catalog entries. Role weights still control how often the composer tries to pick that role; cost controls how many fit after a pick.

Fewer but scarier ice enemies

Catalog: higher ice costs + ice element knobs already tanky. Theme bias toward Ice. Don’t raise budget as much as for physical spam.

Breathers between waves feel too short/long

Tune delayAfterClear only — does not change composition.

Quick mental model Budget × waves ≈ how much “enemy cost” you can buy.
Role/theme weights ≈ which kinds you try to buy.
Catalog costs ≈ price tags.
Block HP/damage ≈ same army, tougher stats.
Max enemies/wave ≈ hard ceiling on density.

11. Debugging

Watch the Unity Console when a combat starts:

[WaveComposer] seed=… block=0 combat=2 theme=Ice Assault waves=3
budgetPerWave=140 spent=392 elite=yes roles=BeamSniper×2, Turret×3, …
  • spent much lower than budgetPerWave × waves → hitting caps early, or expensive picks, or loop stop conditions.
  • spent ≈ cap × cheapest cost → you’re maxed on count; raise budget won’t add bodies until you raise maxEnemiesPerWave.
  • Theme never shows in UI (by design); it only biases element rolls and appears in this log.

12. Limits & gotchas

  • Costs don’t scale with block — only budget and HP/damage do. Past the enemy-count cap, HP/damage are the remaining difficulty levers.
  • Within-block HP/damage are flat — don’t expect combat 3 to be tankier than combat 1 unless you change that design again.
  • Spawn positions aren’t deterministic — same composition, different room spots.
  • Slight budget overspend is allowed so waves aren’t empty when remaining is awkward.
  • Legacy wave assets are unused by the linear RunManager path; branching/map legacy may still reference them.
  • Commit 22 (not done yet) will pre-roll the next block at upgrade so preview/UI can show the same encounters you’ll fight.
